Form Validation

General

Validation provides a means to ensure the user is inputting usable data into a form. Form validation can be expressed to a user initially by denoting required inputs with the required symbol; or, after a user enters invalid information into the form through warnings and invalid states.

Purpose

Provide a method for verifying user input is desired and usable information.

Examples

Pristine input field

Pristine input fields are fields which have not been interacted with. A pristine field that is in an invalid state should not appear to be in an invalid state (i.e. do not display red border, validation messages, or icons). After the user interacts with the input, apply invalid state styles.

Required input field

The required field element should be applied to all required input fields. This element is meant to be used in regular or complex forms that have three or more input fields. Not all forms will need to use the require field notification. For example, a sign in page often includes a form with username and password input fields. These inputs are understood to be required to sign in, but usually don't include a required mark.

A "required" note is not necessary at the bottom of the form since the require asterisk is widely used and understood

Hints

Use the following three methods to instruct the user when a user requires additional information to complete a form field.

1. Inline label hint

Inline label hints must always be placed inside of parenthesis. The hint should be clear, short, and helpful.

2. Validation message after input

Validation messages placed after the input are visible only after the user has interacted with the input. The text should be read and easy to read. Messages should be clear, short, and helpful. For longer messages use the info icon method (see below).

3. Info icon to launch tooltip

If your hint text is too complex for the other methods, use an info icon and tooltip. Clicking/Hovering the info icon displays a tooltip. The tooltip should not contain extensive documentation or instructions. It should contain just enough to instruct the user on how to complete the input.

Form Messages

Form messages are used when portions of the form are not visible to the user (ex: in tabs, accordions, or tile grids). The message should appear at the top of the form and should instruct the user that additional changes are required to complete the form.

Tile Grids

Tile grids hide information about each element in the grid. Form messages make it obvious to the user that a tile in the tile grid requires additional attention. The offending tile should similarly be marked as invalid.

Accordion

Accordions hide information about each section. Form messages make it obvious to the user that a section of the accordion requires additional attention. The offending section should similarly be marked as invalid.

Validating data asynchronously

Sometimes it may be necessary to validate user input on a server. A loader clearly indicates to the user that validation is occurring. When the validation is complete indicate to the user the result of the validation with status icons. In the event that validation fails, clicking on the info icon should present a tooltip with additional information to correct the error.

Asynchronous validation should be de-bounced to allow the user to complete input before validation occurs. This prevents validation from occurring on each keystroke.

Appearance

Common Patterns

  • Use validation as a means of communicating incorrect or missing inputs to the user.
  • Pristine fields are invalid when loading the page, but don't display an invalid state until interacted with by the user.
  • The required symbol should be used when you have three or more input options.
  • Limit the use of input fields to only what is necessary for completing the form. Do not overload the user with input fields.
  • Asynchronous validation provides real-time feedback to the user as validation occurs.
