Dialog

General

Dialogs allow for a system to overtly call the attention of the user from the page to its content. This content can be various information, but it is always something a developer would like the user should do or know. Dialogs are relevant to the page they are on to keep information available to users as they interact with the component. Some usage examples could be a warning message, instructions, or input fields necessary for progression. Essentially, the dialog element calls the attention of the user in order to have a conversation.

Purpose

Call the attention of the user to have a conversation; and, allow for the user to give a response. The user may defer a response by closing the dialog.

Specifications
  • Dialog glass (transparent grey background) covers all other UI elements when dialog is displayed.
  • Dialog glass helps to accentuate the dialog and its content.
  • A Close button will always be an available option within a dialog.
  • The Dialog can be moved to allow the user to read, (through the glass) content underneath.
  • When moving a dialog, the dialog content should also become transparent to allow a user to read content beneath.
  • Make the title element of a dialog the handle for its movement.

Appearance

Common Patterns

  • Communicate with the user information by greying out the page and highlighting its content in the form of a dialog box.
  • Consider the amount of content within the dialog as you don't want to overload the element with too much information.
  • Ask, "Does this make more sense as it's own page?" If yes then it probably shouldn't be in a dialog.
  • Allow for a user to view and edit information in an object's details.
  • A confirmation for ensuring a user wants to delete an object.
  • Simple forms with a few inputs. These inputs can be form fields, buttons, etc.
  • Always allow for a user to close the dialog with a close button, (ias-dialog-cancel-button).
  • Dialog content should initially be placed near the center of the page, only slightly offset above the middle to create the golden ration.

Nesting

Nesting Dialogues in other Elements

Dialogs should NEVER be nested into other elements. Dialogs are fixed to the window, and grey out all other elements to call the attention of the user to the content the dialog holds.

Nestable Components

Dialogs can nest nearly any component to help convey the message to a user or get necessary information. The list below are likely to be common components, but are not exhaustive.

  • ias-icon-button
  • ias-button
  • ias-dialog-content (paragraph)
  • input (Form fields)

Responsive Displays

As a screen is scaled down the div that holds the dialog glass and dialog content will be scaled down as well. In doing this will also scale down the dialog content along with its content.

Dialog Glass

  • A transparent grey background that covers the original page to help accentuate the dialog content.
  • A dialog can be moved around in the window by the title element to see the content beneath the dialog glass.
  • Moving a dialog causes the dialog to become transparent, allow a user to easily read content beneath.
  • Closing a dialog should be done by clicking the exit icon button. A developer can choose to have a click on the glass close the window similar to a menu.
  • The dialog glass will absorb mouse events when present.

Dimensions

The dimensions of the dialog should grow to the content within the dialog.

Tab Support

Everything else is disabled. Tab support should begin with content and end with 'X'
